# RAG 与 LLM Wiki 方法论的对比分析

> 本分析基于[[wiki/sources/rag-practical-guide|RAG 全流程深度实战指南]]与[[wiki/sources/llm-wiki-idea|LLM Wiki 思想文档]]，梳理两种知识管理范式的根本差异。

## 核心分歧：查询时编译 vs 摄取时编译

两种范式的根本差异在于 **知识处理的时机**：

| 维度 | 传统 RAG | LLM Wiki 方法论 |
|------|---------|----------------|
| **知识处理时机** | ⏱ 查询时实时检索 | 📦 摄取时预先编译 |
| **知识存储形式** | 向量数据库（稠密向量） | Markdown 维基页面（结构化文本） |
| **知识积累** | ❌ 每次查询重新检索，无持久积累 | ✅ 知识被编译一次并持续更新 |
| **人类参与度** | 低（自动化 pipeline） | 高（人类策展 + LLM 维护） |
| **查询延迟** | 中等（检索 + 生成） | 低（LLM 直接阅读编译好的维基） |
| **可解释性** | 中等（可溯源但知识分散在向量中） | 高（维基页面直接可读、可编辑） |
| **更新方式** | 重新索引文档即可 | LLM 读取新源后增量更新相关页面 |
| **规模伸缩** | ⭐⭐⭐⭐⭐ 海量数据（百万级文档） | ⭐⭐⭐ 中小规模（精耕细作） |
| **知识精确度** | ⭐⭐⭐ 依赖检索质量，可能召回噪声 | ⭐⭐⭐⭐⭐ 人类策展 + LLM 维护 |
| **跨文档推理** | ⭐⭐ 需多跳检索或 Graph RAG | ⭐⭐⭐⭐ 维基天然聚合多源信息 |

## 各自优势

### 传统 RAG 的优势
1. **海量数据友好** — 百万级文档无需人工预处理
2. **实时更新** — 新文档加入后即时可用
3. **低维护成本** — pipeline 自动化运行，无需人工策展
4. **细粒度检索** — 可精确定位到文档片段级别

### LLM Wiki 的优势
1. **知识沉淀** — 每次交互都在积累，而非从头检索
2. **人类可控** — 维基页面可直接阅读、修改、策展
3. **跨源综合** — 天然聚合多篇文章的观点，发现矛盾
4. **低查询成本** — 无需每次检索和重排序，减少 token 消耗
5. **高质量输出** — 编译后的维基经过整合，质量更稳定

## 适用场景决策

```
                    数据量大？
                    ┌───┐
                    │   │
              ┌─────┴───┴─────┐
              ↓               ↓
            是/              否/
        实时性要求高         追求知识深度
              ↓               ↓
        ┌──────────┐    ┌──────────┐
        │ 传统 RAG  │    │ LLM Wiki │
        │ (海量检索) │    │ (精耕细作)│
        └──────────┘    └──────────┘
```

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 企业全量文档问答（10 万+） | 传统 RAG | 人力无法逐篇策展 |
| 个人知识库（< 1000 篇） | LLM Wiki | 质量优先，人工可控 |
| 实时数据（新闻、股价） | 传统 RAG | 需要分钟级更新 |
| 深度研究（学术、分析） | LLM Wiki | 需跨源综合与矛盾发现 |
| 客户服务知识库 | 传统 RAG | 海量 FAQ + 实时更新 |
| 技术笔记与博客 | LLM Wiki | 个人策展，长期积累 |

## 混合可能

两者并非完全互斥，可以结合使用：

```
原始数据 → RAG 快速检索 ─→ LLM 整理 → 更新维基页面 → 维基作为精炼知识源
                ↑                                              │
                └──────────────────────────────────────────────┘
```

- **RAG 作为"草稿"层**：从海量文档中快速检索相关片段
- **LLM Wiki 作为"精炼"层**：将检索结果整理为结构化维基页面
- **查询时**：优先查询维基（高质量），维基覆盖不足时回退到 RAG 检索

这种混合架构兼顾了 RAG 的海量覆盖与 LLM Wiki 的知识沉淀。
