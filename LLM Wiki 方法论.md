# LLM Wiki 方法论

一种利用 LLM 增量构建和维护个人知识库的模式。

> 本文是一份思想文档，设计用于复制粘贴给你的 LLM 智能体（如 OpenAI Codex、Claude Code、OpenCode/Pi 等）。它的目标是传达核心理念，而你的智能体会与你协作，细化为具体实现。

## 核心理念

大多数人使用 LLM 处理文档的方式是 **RAG（检索增强生成）**：上传文件集合，LLM 在查询时检索相关片段并生成答案。这种方式有效，但 LLM 每次都在从零重新发现知识——没有积累。问一个需要综合五份文档的微妙问题时，LLM 必须每次重新查找和拼凑相关片段。没有任何东西被构建起来。NotebookLM、ChatGPT 文件上传和大多数 RAG 系统都是这样工作的。

这里的思路不同。LLM **不是**在查询时才从原始文档中检索，而是**增量构建和维护一个持久的维基**——一个结构化的、相互链接的 markdown 文件集合，位于你和原始源之间。当你添加新源时，LLM 不只是将其编入索引以供日后检索。它会读取源，提取关键信息，并将其整合到现有的维基中——更新实体页面、修订主题摘要、标注新数据与旧声明之间的矛盾、加强或挑战不断演进中的综合结论。**知识被编译一次，然后持续更新**，而不是每次查询时重新推导。

这是关键区别：**维基是一个持久的、不断积累的制品。** 交叉引用已经在那里。矛盾已经被标记。综合已经反映了你读过的所有内容。随着你添加的每个源和提出的每个问题，维基变得越来越丰富。

你从不需要（或很少需要）自己编写维基——LLM 编写和维护所有内容。你负责 sourcing、探索和提出正确的问题。LLM 完成所有琐碎工作——总结、交叉引用、归档和簿记，这些正是一个知识库随时间推移真正有用的关键。实际使用时，我在一边打开 LLM 智能体，另一边打开 Obsidian。LLM 根据我们的对话进行编辑，我实时浏览结果——点击链接、查看图谱视图、阅读更新后的页面。Obsidian 是 IDE，LLM 是程序员，维基是代码库。

这种方法可以适用于许多不同的场景。几个例子：

- **个人成长**：追踪你的目标、健康、心理学、自我提升——归档日记条目、文章、播客笔记，随时间构建一幅关于你自己的结构化画像。
- **研究**：在几周或几个月内深入研究一个主题——阅读论文、文章、报告，逐步构建一个包含不断演进的论点的综合维基。
- **读书**：边读边归档每个章节，为人物、主题、情节线索及其关联构建页面。读完后你就拥有了一个丰富的伴读维基。想象一下像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝维基——由志愿者社区多年构建的数千个相互链接的页面，涵盖人物、地点、事件、语言。你可以在阅读时个人构建类似的东西，LLM 做所有的交叉引用和维护工作。
- **商业/团队**：由 LLM 维护的内部维基，输入源来自 Slack 线程、会议记录、项目文档、客户通话。可能让人参与审查更新。维基保持最新，因为 LLM 做了团队中没人想做的维护工作。
- **竞争分析、尽职调查、旅行规划、课程笔记、爱好深潜**——任何你随时间积累知识并希望有组织地存放的场景。

## 架构

分为三个层次：

**原始源**——你精心挑选的源文档集。文章、论文、图片、数据文件。这些是不可变的——LLM 从中读取但从不修改。这是你的真相来源。

**维基**——一个由 LLM 生成的 markdown 文件目录。包含摘要、实体页面、概念页面、比较分析、概览、综合综述。LLM 完全拥有这一层。它创建页面，在新源到来时更新它们，维护交叉引用，保持一切一致。你阅读；LLM 编写。

**模式**——一个文档（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM 维基的结构、惯例以及在摄取源、回答问题或维护维基时应遵循的工作流。这是关键的配置文件——它让 LLM 成为一个有纪律的维基维护者，而不是一个通用的聊天机器人。你和 LLM 会随着时间共同演进这个文档，找出适合你领域的最佳实践。

## 操作

**摄取（Ingest）**。你将一个新源放入原始集合，告诉 LLM 处理它。一个典型的流程：LLM 读取源，与你讨论关键要点，在维基中写一个摘要页面，更新索引，更新维基中相关的实体和概念页面，并在日志中追加一条记录。一个源可能触及 10-15 个维基页面。我个人偏好一次摄取一个源并保持参与——我阅读摘要，检查更新，引导 LLM 关注重点。但你也可以一次批量摄取多个源，减少监督。由你发展出适合自己风格的工作流，并将其记录在模式中供未来会话使用。

**查询（Query）**。你向维基提问。LLM 搜索相关页面，阅读它们，并合成带引用的答案。答案可以根据问题类型采用不同形式——markdown 页面、对比表格、幻灯片（Marp）、图表（matplotlib）、画布（canvas）。一个重要的洞察：**好的答案可以归档回维基作为新页面。** 你要求做的比较分析、深度分析、发现的联系——这些都是有价值的，不应该消失在聊天历史中。这样，你的探索就像摄入的源一样，在知识库中不断积累。

**检查（Lint）**。定期要求 LLM 对维基进行健康检查。检查：页面之间的矛盾、被新源取代的陈旧声明、没有入链的孤儿页面、被提及但缺少独立页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 善于建议新的待研究问题和待寻找的新源。这能保持维基在增长过程中的健康。

## 索引与日志

两个特殊文件帮助 LLM（和你）在维基增长时导航。它们有不同的用途：

**`index.md`** 是面向内容的。它是维基中所有内容的目录——每个页面列出名称和链接、一句话摘要，以及可选的元数据如日期或源数量。按类别组织（实体、概念、源等）。LLM 在每次摄入时更新它。回答查询时，LLM 先读取索引定位相关页面，然后深入阅读。这在中等规模（约 100 个源、约数百个页面）下效果出奇的好，避免了基于嵌入的 RAG 基础设施的需要。

**`log.md`** 是按时间顺序的。它是一个只追加的记录，记录了什么时间发生了什么——摄入、查询、检查。一个有用的技巧：如果每条记录以一致的前缀开头（例如 `## [2026-04-02] ingest | Article Title`），日志就可以用简单的 Unix 工具解析——`grep "^## \[" log.md | tail -5` 给出最近 5 条记录。日志让你了解维基的演变时间线，帮助 LLM 理解最近完成了什么。

## 可选：CLI 工具

在某个时候，你可能想构建一些小工具，帮助 LLM 更高效地操作维基。最明显的是一个维基页面搜索引擎——小规模时索引文件就足够了，但随着维基增长，你需要真正的搜索功能。[[qmd|qmd]] 是一个不错的选择：它是一个针对 markdown 文件的本地搜索引擎，具有混合 BM25/向量搜索和 LLM 重排序功能，全部在本地运行。它既有 CLI（LLM 可以 shell 调用），也有 MCP 服务器（LLM 可以将其作为原生工具使用）。你也可以自己构建更简单的方案——LLM 可以帮助你即时编码一个简单的搜索脚本。

## 技巧与建议

- **Obsidian Web Clipper** 是一个浏览器扩展，可将网页文章转换为 markdown。非常适用于快速将源加入原始集合。
- **本地下载图片。** 在 Obsidian 设置 → 文件和链接中，将"附件文件夹路径"设置为固定目录（如 `raw/assets/`）。然后在设置 → 快捷键中，搜索"Download"找到"Download attachments for current file"并绑定快捷键（如 Ctrl+Shift+D）。剪藏文章后，按下快捷键，所有图片就会下载到本地磁盘。这是可选的，但很有用——它让 LLM 可以直接查看和引用图片，而不依赖于可能失效的 URL。注意 LLM 无法在一次读取中直接解析 markdown 中的内嵌图片——解决方法是用 LLM 先阅读文本，然后单独查看部分或所有引用的图片以获取更多上下文。这有点笨拙，但效果不错。
- **Obsidian 的图视图**是查看维基形态的最佳方式——哪些页面相连，哪些是枢纽，哪些是孤儿。
- **Marp** 是一种基于 markdown 的幻灯片格式。Obsidian 有对应的插件。适用于直接从维基内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可以对页面 frontmatter 执行查询。如果 LLM 为维基页面添加了 YAML frontmatter（标签、日期、源数量），Dataview 可以生成动态表格和列表。
- 维基只是一个 markdown 文件的 git 仓库。你可以免费获得版本历史、分支和协作能力。

## 为什么这有效

维护知识库中最繁琐的部分不是阅读或思考——而是**簿记**。更新交叉引用、保持摘要及时、标注新数据与旧声明之间的矛盾、在几十个页面间保持一致性。人类放弃维基是因为维护负担的增长速度超过了价值增长。LLM 不会感到无聊，不会忘记更新交叉引用，可以一次触及 15 个文件。维基保持维护状态，因为维护成本几乎为零。

人类的工作是策划源、指导分析、提出好问题、思考这一切意味着什么。LLM 的工作是其他所有事。

这个想法在精神上与 Vannevar Bush 的 Memex（1945 年）相关——一个个人化、经过策划的知识库，文档之间有联想路径。Bush 的愿景比后来的网络更接近这个模式：私有的、主动策划的，文档之间的连接与文档本身同样重要。他无法解决的问题是谁来做维护工作。**LLM 解决了这个问题。**

## 附注

本文故意保持抽象。它描述的是理念，而不是具体的实现。确切的目录结构、模式约定、页面格式、工具——所有这些都取决于你的领域、你的偏好和你选择的 LLM。上述一切都是可选和模块化的——选择有用的，忽略不需要的。例如：你的源可能只有文本，所以你不需要图片处理。你的维基可能足够小，索引文件就足够了，不需要搜索引擎。你可能不关心幻灯片，只想要 markdown 页面。你可能想要一套完全不同的输出格式。使用本文的正确方式是与你的 LLM 智能体分享，并一起协作实例化一个适合你需求的版本。本文的唯一工作是传达这个模式。你的 LLM 可以搞定其余部分。
